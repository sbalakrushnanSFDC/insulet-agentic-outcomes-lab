### E2E-01: Happy Path — SF Consent Event → MC Double Opt-In SMS

| Field | Value |
|---|---|
| ID | `[NGASIM-3336]_E2E01_200 Happy Path_Full chain consent to SMS send_(E2E)` |
| ACs | AC-01, AC-04, AC-05, AC-06 |
| Preconditions | NGASIM-3058 trigger deployed to SF sandbox; all 3 MuleSoft layers running on CloudHub; MC sandbox with SMS Definition configured; SF Named Credential for MuleSoft configured |

**Steps:**

1. **Given** a Contact record exists in SF sandbox with Phone = +15551234567
2. **And** a ContactPointTypeConsent record linked to the Contact with ConsentType = SMS
3. **When** the ConsentStatus field is updated to 'Opt In Pending' via SF UI or API
4. **Then** the Apex trigger (NGASIM-3058) publishes a ContactPointTypeConsentChangeEvent
5. **And** MuleSoft exp-api receives the event via SF Named Credential
6. **And** exp-api validates mandatory fields (ConsentId, ConsentStatus, Phone, ConsentType, ContactId, ConsentTimestamp)
7. **And** exp-api transforms the SF payload to MC format via DataWeave
8. **And** exp-api routes to prc-api with X-Correlation-ID header
9. **And** prc-api processes and routes to sys-api
10. **And** sys-api calls MC Classic REST API to trigger SMS send
11. **Then** the patient receives a double opt-in confirmation SMS at +15551234567
12. **And** exp-api returns 202 Accepted with correlationId in response body

**Verification Queries:**

```sql
-- SF: Verify ContactPointTypeConsent exists
SELECT Id, Name, PartyId, DataUsePurpose, CaptureDate,
       PrivacyConsentStatus, ContactPointType
FROM ContactPointTypeConsent
WHERE ContactPointType = 'Phone'
  AND PrivacyConsentStatus = 'OptIn'
ORDER BY CaptureDate DESC LIMIT 5

-- SF: Verify platform event was published (via EventBusSubscriber)
SELECT Id, EventUuid, CreatedDate, CreatedById
FROM ContactPointTypeConsentChangeEvent
ORDER BY CreatedDate DESC LIMIT 5
```

**Expected Result:** SMS delivered within 30 seconds of consent status change. Correlation ID traceable from exp-api response through prc-api and sys-api logs to MC delivery receipt.

### E2E-07: DLQ Routing After Exhausted Retries

| Field | Value |
|---|---|
| ID | `[NGASIM-3336]_E2E07_500 DLQ_Retries exhausted routes to DLQ with envelope_(Integration)` |
| ACs | AC-08 |
| Preconditions | exp-api deployed to CloudHub; Anypoint MQ DLQ queue created; prc-api intentionally unavailable (stopped) |

**Steps:**

1. **Given** prc-api is unreachable (stopped or returning 500)
2. **When** POST a valid consent event to exp-api
3. **Then** exp-api attempts 3 retries with exponential backoff (2s, 4s, 8s)
4. **And** after all retries exhausted, exp-api publishes to `exp-sms-consent-dlq`
5. **And** DLQ message contains structured envelope:
   - `dlqMetadata.correlationId` matches response correlationId
   - `dlqMetadata.failureReason` contains HTTP error description
   - `dlqMetadata.retryCount` = 3
   - `dlqMetadata.lastFailureTimestamp` is recent
   - `dlqMetadata.targetEndpoint` is the prc-api URL
   - `originalPayload.Phone` is masked (e.g., `+155***4567`)
6. **And** exp-api returns 202 Accepted (event accepted into DLQ for deferred processing)

**Verification:**
```bash
# Check Anypoint MQ DLQ depth via Anypoint CLI
anypoint-cli-v4 runtime-mgr:anypoint-mq:message:browse \
  --environment=sandbox \
  --region=us-east-1 \
  --queue=exp-sms-consent-dlq
```

### E2E-11: Circuit Breaker — Sustained Failure Opens Circuit

| Field | Value |
|---|---|
| ID | `[NGASIM-3336]_E2E11_503 Circuit Open_Sustained failure trips circuit breaker_(Integration)` |
| ACs | AC-15 |
| Preconditions | exp-api deployed to CloudHub; prc-api returning 503 for all requests |

**Steps:**

1. **Given** prc-api is returning 503 Service Unavailable consistently
2. **When** 10+ consent events are submitted to exp-api over 5 minutes
3. **Then** after the failure threshold (5 consecutive failures), the circuit breaker opens
4. **And** subsequent requests fast-fail with 503 without attempting prc-api call
5. **And** response body includes `{ errorCode: "CIRCUIT_OPEN", message: "...temporarily unavailable..." }`
6. **And** all fast-failed events are routed to DLQ
7. **When** prc-api recovers (returns 200)
8. **And** circuit breaker timeout (300 seconds) elapses
9. **Then** circuit transitions to HALF_OPEN
10. **And** next request is attempted against prc-api
11. **If** prc-api succeeds, circuit CLOSES and normal processing resumes

| Field | Value |
|---|---|
| ID | `[NGASIM-3336]_E2E13_200 Replay_Events replayed after exp-api downtime recovery_(E2E)` |
| ACs | AC-01, AC-08 |
| Preconditions | NGASIM-3058 trigger deployed; exp-api deployed with replay ID persistence; SF sandbox accessible |

**Steps:**

1. **Given** exp-api is running and consuming platform events from SF
2. **When** exp-api is stopped (simulating CloudHub deployment or outage)
3. **And** 5 consent events are published in SF during the downtime window
4. **When** exp-api is restarted
5. **Then** exp-api reconnects to SF platform event bus with the last persisted replay ID
6. **And** all 5 missed events are replayed and processed
7. **And** idempotency check prevents any events that were processed before the downtime from being duplicated

**Verification:**
- Count of processed events before downtime + replayed events after recovery = total events published
- No duplicate SMS sends (idempotency store prevents double-processing)

## Handoff to SECURITY Agent

The SECURITY agent should:

1. Review exp-api authentication model: how does SF authenticate to exp-api? (Named Credential, OAuth, mTLS). Verify that unauthenticated requests are rejected.
2. Assess PII handling across the entire flow: phone number masking in logs, DLQ, error responses. Verify TCPA compliance for SMS consent data.
3. Review OAuth2 client credentials configuration for MuleSoft-to-prc-api and MuleSoft-to-MC. Verify token storage is secure (not in logs, not in ObjectStore).
4. Evaluate TLS configuration: is TLS 1.2+ enforced on all HTTP connections (inbound and outbound)?
5. Assess ObjectStore security: is the idempotency store accessible only to the exp-api application? Can other CloudHub apps read idempotency keys?
6. Review DLQ message security: are DLQ messages encrypted at rest? Who has access to browse the DLQ?
7. Evaluate consent data handling under TCPA and GDPR: is consent data (opt-in status, phone number, timestamp) handled in accordance with regulatory requirements?
8. Assess cross-cloud identity alignment: ContactId → SubscriberKey mapping must be consistent. Verify no identity leakage or mismatch.
9. Review circuit breaker and retry behavior for timing attacks: can an attacker exploit retry delays to infer system state?
10. Verify that error responses do not leak internal implementation details (stack traces, internal URLs, ObjectStore keys).
